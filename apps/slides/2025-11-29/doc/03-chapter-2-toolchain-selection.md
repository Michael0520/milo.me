# 第二章：工具鏈選型 - 決策的藝術

## 時間分配

9-11 分鐘（核心章節）

## 內容大綱

### 1. 工具鏈演進：從能用到好用 (2 分鐘)

- **Lerna (2016~)**：老牌王者的遺產（現況：仍有人使用，但已非主流）
  - 專注於多套件發布流程
  - 效能已無法滿足現代大型專案需求
  - Note: v5+ 由 Nx 團隊接管管理，v6+ 任務調度委託給 Nx 運行器
  - 建議：新專案直接選 Nx 或 Turborepo
- **pnpm Workspaces (2018~)**：極致的依賴管理
  - 透過 pnpm-workspace.yaml 和 catalog 協議
  - 實現高效的依賴管理與空間節省
  - 70% less disk space
- **Intelligent Build Systems (2020~)**：智慧型構建系統
  - Nx 與 Turborepo 不僅是任務執行器
  - 更是智慧型構建協調器
  - 實現精確依賴追蹤與任務編排
  - Remote Caching、Task Orchestration

### 2. 智慧對決：Nx vs Turborepo (5-6 分鐘)

- **快取機制**
  - Nx：本地與遠端快取（Nx Cloud 或自托管）
  - Turborepo：本地與遠端快取（Vercel Remote Cache）
- **任務編排**
  - Nx：強大的任務編排（dependsOn + 進階過濾）
  - Turborepo：自動編排（^build 語法）
- **依賴視覺化**
  - Nx：互動式圖表（nx graph）
  - Turborepo：靜態圖表（Graphviz）
- **程式碼生成**
  - Nx：生成器（nx generate）
  - Turborepo：不支援（需外部工具）
- **插件生態**
  - Nx：豐富插件（React, Angular, Node）
  - Turborepo：生態較少（專注核心功能）
- **選擇建議**
  - Nx：功能全面的開發平台，適合需要深度整合的團隊
  - Turborepo：極致效率的構建系統，適合追求簡潔配置的團隊

### 3. 決策樹：選擇適合的工具 (2 分鐘)

- **小型團隊 (< 10人)**
  - 需要快速啟動 → Turborepo
- **大型團隊 (> 10人)**
  - 需要程式碼生成 → Nx
  - 混合方案 → pnpm + Turbo
- **考慮因素**
  - 現有技術棧整合度
  - 學習曲線 vs 功能需求
  - 未來擴展性需求

## 完整逐字稿

剛才我們談到 Monorepo 是「基礎設施決策」，那接下來問題來了：既然決定走 Monorepo，工具鏈要怎麼選、怎麼落地？

這就是第二章要談的：**工具鏈選型**。

工具鏈的演進，可以說是從「能用」到「好用」的過程。

最早談到 Monorepo，大家一定會搜尋到 **Lerna**。它是老牌王者，專注在多套件發布流程，但效能已經跟不上現代大型專案。

雖然 Lerna 現在還有人在使用，但隨著 Nx 和 Turborepo 的出現，大多數新專案已經轉向這些更現代化的工具。從 Lerna v5 開始，Nx 團隊接管了 Lerna 的管理工作；到了 v6+，Lerna 的任務調度直接委託給 Nx 的任務運行器，所以 `lerna run` 現在可以享受 Nx 的快取和命令分發功能，效能大幅提升。

但如果你要開始新專案，我會建議直接選 Nx 或 Turborepo，不用再繞一圈。

接著是 **pnpm Workspaces**。透過 `pnpm-workspace.yaml` 和 catalog 協議，實現了高效的依賴管理與統一版本策略。最直接的感受是：硬碟空間省了 70%，設定檔案也少寫很多。這對開發體驗來說是很大的提升。

然後是智慧型構建系統的時代：**Nx** 和 **Turborepo**。  
它們不只是任務執行器，而是能理解依賴圖譜、實現智能快取與任務編排的協調器。Vercel 推出了 Turborepo，Nx 後來也把 Lerna 整合進來。兩者都能做到遠端快取、任務編排這些進階功能。

我個人很喜歡 Nx 的 plugin 系統，它可以讓專案結構客製化，還能做到自動化工具生成。這對大型團隊來說特別有用。

接下來我們來比較目前最主流的兩個工具：**Nx 跟 Turborepo**。

先說**快取機制**。兩者都有本地與遠端快取。Nx 搭配 Nx Cloud（或自托管方案），可以做到分散式快取，讓團隊共享建置結果。Turborepo 基於 Rust 核心，速度非常快，Vercel Remote Cache 也提供類似的遠端快取服務。這部分兩者都很強。

**任務編排**這裡差異比較明顯。Nx 有強大的 `dependsOn` 機制，可以精確控制執行順序與並行度，還能做進階過濾。Turborepo 用 `^build` 這種簡單直觀的語法，自動處理依賴關係。對大多數專案來說已經足夠，但如果你需要複雜的客製化，就得另外寫配置檔案，不像 Nx 有原生 API 支援。

**依賴視覺化**這點 Nx 優勢很明顯。Nx 的 `nx graph` 是互動式圖表，可以快速瀏覽、篩選、深入分析依賴關係。這在 debug 或重構時非常有用。Turborepo 支援用 Graphviz 產生靜態圖檔，但沒有互動性。如果依賴循環比較深，整張圖打開會很難追查；Nx 可以循序漸進地追蹤，從上層一路看到底層。

**程式碼生成**方面，Nx 有內建的 code generation plugin。用 `nx generate` 可以快速搭建 APP 或函式庫，確保結構一致，不會漏掉 `package.json` 或設定檔案。Turborepo 沒有這個功能，需要靠外部工具。

**插件生態**的話，Nx 出來比較久，官方與社群插件豐富，深度整合 React、Angular、Node 等主流框架。Turborepo 生態相對較少，但更專注在核心構建功能。

所以**選擇建議**是什麼？

如果你的團隊規模較大，或產品線完整、需要深度整合，我會建議用 **Nx**。功能全面，但學習成本會高一些，適合需要架構約束性的大型企業。

如果追求快速啟動、簡潔配置，**Turborepo** 是很好的選擇。尤其在 JavaScript/TypeScript 整合上非常順暢，速度也快。

兩套工具各有優勢，可以根據團隊需求選擇。

工具選好了，接下來就是團隊治理的問題：版本策略要怎麼訂？權限管理怎麼做？遠端快取怎麼設定？這些我們在下一章會詳細討論。
