第一波 — Rust 化：為什麼工具都在重寫
feedback loop 的長度決定 DX 的上限——速度不是奢侈品，是設計前提
  • • Oxlint：ESLint 數十秒 → 毫秒（官方 benchmark 4–16x），lint 從 code review 提前到打字當下
  • • Rolldown：10–30x faster builds、記憶體下降 100x，Vite 8 已預設、plugin 完整相容
  • • 對 Vue 開發者的實感：HMR、CI、lint 三條 feedback loop 同時縮短
  • • 實戰：Oxlint import/no-cycle 即時抓循環依賴（⚠️ TS path alias 邊際情況 #20551）
  • • 設計邏輯：工具快到「即時」之後，整類工作流程（lint-staged、pre-commit cache）直接消失

第二波 — 統一化：NX vs Vite 生態的路線之爭
兩條路線都在解「工具碎片化」，但哲學相反——平台 vs 組合
NX 路線 — 一個平台管所有事
project graph、task caching、generators、CI 整合全包

Vite 生態路線 — 可組合的單一用途工具
pnpm workspaces + semantic catalogs（catalog:build / catalog:lint，antfu 模式）
vite-plus vp run -r：依 dep order 執行 + content-based task caching
dependency-cruiser 作為 NX Graph 的輕量替代
立場與選型結論
Vue / Vite / Nuxt 核心貢獻者（含 antfu）都不用 NX——「對 Vue/Nuxt 專案來說是過度設計」
Vue/Nuxt monorepo（<10 packages）→ pnpm + Oxlint + dependency-cruiser 已夠

第三波 — AI 化：infra 的新使用者是 agent
AI 改一處壞三處？那不是 AI 的問題，是 infra 的問題
  • • 糾纏的 module graph = AI 的不確定性放大器——同一個根因也讓 tree-shaking 不可預測、chunk 膨脹、LCP 惡化
  • • 解法是機器可驗證的邊界鏈：AI 改 A → Oxlint 確認無 cycle → TS 抓型別 → build → CI
  • • CLAUDE.md / AGENTS.md = 把 boundary 規則寫成 AI 看得懂的格式（antfu 也在做這件事）
  • • 整個 Vite 生態（Cloudflare 收購後）的方向：讓 AI agent 成為工具的一等公民

NX 免費功能詳解

免費功能（無需 NX Cloud）

Free Tier
  • 🗂
    nx graph — 互動式 dependency graph
    可匯出 HTML（--file=output.html）、JSON。支援 focus 模式、dependency 追蹤
  • ⚡
    Local Task Caching — .nx/cache
    相同 input → 從 cache 還原，幾乎瞬間完成。同一台機器有效
  • 🎯
    nx affected — 只跑受影響的 package
    Git-based change detection → project graph → 遞迴找下游依賴。lockfile 改動預設標記全部受影響
  • 🛡
    @nx/enforce-module-boundaries
    ESLint rule，用 tag system 做 cross-project boundary enforcement。可設定 sourceTag / onlyDependOnLibsWithTags
  • 🔧
    Code Generation
    generators + executors，AST-level 程式碼修改
  • 🟢
    @nx/nuxt + @nx/vue 官方 Plugin
    2024-02 釋出，支援 Nuxt 3.x，整合 build / test / serve

NX Cloud（付費）

Paid Only
  • ☁️
    Distributed Remote Cache
    跨機器、跨 CI 共享 cache。沒有這個，CI 每次都重跑（大型 repo 差距可達 3 分鐘 → 30 分鐘）
  • 🔀
    Distributed Task Execution
    自動分配 task 到多台 CI agent，NX 自己做負載均衡
  • 📊
    Insights & Analytics
    CI 效能分析、cache hit rate 追蹤
  • ⚠️
    2026 重大改動
    移除免費自架 remote cache plugin（安全風險理由）。現在 remote cache = NX Cloud only
結論： 中小型團隊 / 個人 repo，不需要 NX Cloud。 Local cache + affected 免費就夠，遠端分散式 cache 只有大型分散團隊才划算。

Vite 生態的立場 — antfu & 核心團隊 Q&A 強力支持論點

核心觀點：Vue 生態根本不用 NX

Ecosystem Signal
Vue / Vite / Nuxt 的核心貢獻者（包含 Anthony Fu / antfu）都用 pnpm workspace，沒有人在用 NX。 這不是偶然 — 而是生態的默契：輕量、可組合的工具 > 重型框架。 antfu 的哲學：每個工具只做一件事，composable，不綁定，可替換。
主要開源 repo 的 monorepo 工具
Repo pnpm NX
vuejs/core ✓ ✗
vitejs/vite ✓ ✗
nuxt/nuxt ✓ ✗
antfu/eslint-config ✓ ✗
antfu 建的 pnpm 輔助工具（不是 NX 替代品，是加強層）
  • eslint-plugin-pnpm
    enforce catalog usage — 讓 ESLint 抓到手動寫版本號的 dep（應改用 catalog:）
  • taze
    輕量 dep update tool，互動式選擇要升的版本
  • ni
    package manager agnostic wrapper（ni = install、nr = run）
  • @antfu/eslint-config
    eslint-plugin-import-x 做 workspace-aware import rules，不需要 NX

antfu：「Categorize Your Dependencies」

antfu.me/posts/categorize-deps
antfu 的文章推廣 semantic catalog 命名模式 — 把依賴按用途分類，讓 pnpm-workspace.yaml 本身就是文件。這正是你的 repo（milo-vite-plus）採用的設計。
catalog:（default）
Production shared deps — Vue, React, Zod…
catalog:build
Build tools — vite, rollup, tsup, vite-plus
catalog:lint
Linting — oxlint, eslint, prettier
catalog:test
Testing — vitest, @testing-library
catalog:script
Dev scripts — tsx, jiti, zx
catalog:preview
Experimental — TS 6 beta、@types/node 25
你的 repo 就是活的範例： Q&A 時可現場開 pnpm-workspace.yaml 給聽眾看 semantic catalog 設計。
Q&A 強力論點： 「如果連 Vite / Nuxt 的核心維護者都用 pnpm workspace + eslint rules 而不是 NX， 代表這套組合確實夠用。NX 適合更大規模的 enterprise monorepo， 但對大多數 Vue/Nuxt 專案來說是過度設計。」

dependency-cruiser — NX Graph 的輕量替代

功能與定位

6,733 stars · 活躍維護
  • 🗺
    Dependency graph 視覺化（HTML / SVG 輸出）
  • 🚫
    Circular dependency 偵測
  • 📐
    Regex-based boundary rules（path patterns）
    類似 NX tag system，但用 file path 定義，不需要額外設定 tags
  • ⚡
    輕量，不需要 daemon 或 cloud
  • 🟢
    Vue / Vite 完整支援

範例設定

// .dependency-cruiserrc.json
{
  "forbidden": [
    {
      "name": "no-circular",
      "severity": "error",
      "from": {},
      "to": { "circular": true }
    },
    {
      "name": "no-app-cross-import",
      "severity": "error",
      "from": { "path": "^apps/([^/]+)" },
      "to": { "path": "^apps/(?!\\1)" }
    },
    {
      "name": "no-packages-to-apps",
      "severity": "error",
      "from": { "path": "^packages/" },
      "to": { "path": "^apps/" }
    }
  ]
}
npx depcruise --validate .dependency-cruiserrc.json apps/
npx depcruise --output-type html --output-to report.html apps/
何時選 dependency-cruiser vs NX： 中小型 Vue/Nuxt 專案（<10 packages）→ dependency-cruiser + Oxc 已夠。 大型 monorepo（10+ packages，需要 tag-based governance + CI affected）→ NX Free。

Q&A 風險評估 Talk 專用

✓ 安全可講（自己親身用過，可當場開 code）

  • pnpm workspace 結構 / semantic catalogs — 這個 portfolio repo 就是活的例子
  • 循環依賴 → chunk size → Web Vitals 的因果鏈 — 邏輯清楚，可推論
  • CLAUDE.md / AGENTS.md 作為 machine-readable context — 自己有用
  • dependency-cruiser boundary rules — 設定檔可展示
  • 「過去 infra 為人設計 vs. 現代 infra 為 AI agent 設計」的框架論述

△ 可講，但要加免責聲明

  • Oxc no-cycle — 說「已實作，但 TypeScript path alias 有邊際情況，使用前要測試」
  • Rolldown / Vite 8 效能 — 說「Vite 8 已使用 Rolldown（2026-03），官方說 10–30x faster builds」，強調「官方數字，未自測」
  • NX 免費功能邊界 — 說「目前本地 cache / affected / graph 免費」，加「NX 版本可能調整」
  • Oxc 比 ESLint 快 — 說「顯著更快（官方 benchmark 4–16x）」，不說 100x
  • Cloudflare 收購 VoidZero — 可提，說「最新消息（2026-06-05）」，但說明工具仍 open-source
  • vite-plus task caching vs NX affected — 要說清楚：vp run -r 跑全部 package，content-based cache 讓未改的瞬間完成（replay），這跟 NX affected 直接跳過不同。混淆了容易被追問

✗ 高風險，完全不提或跳過

  • NX Cloud 具體定價 — 不提，pricing 可能變
  • NX vs Turbo 哪個更好 — 不評論，說「各有定位」
  • 具體效能數字（非自己測的）— 一律說「官方數字」或不說
  • eslint-plugin-vue 在 Oxlint 的相容性細節 — 複雜，容易說錯
  • Cloudflare 收購後的具體 roadmap — 尚未官方確認
  • vite-plus 是否有 remote cache — 尚不確定，不提
萬用收尾句（被問到不熟的問題）：
「這個細節我自己的 repo 沒有實際跑過，我不想在這邊亂說。建議你直接看官方的 changelog 或 benchmark，比我的記憶準確。」

使用情境建議

你的情況（milo-vite-plus 規模）應該選什麼？

中小型 Vue/Nuxt monorepo（<10 packages）

  • ✓ pnpm workspaces（結構即邊界）
  • ✓ Oxlint no-cycle（快速偵測）
  • ✓ dependency-cruiser（graph 視覺化）

  • ✓ NX Free（affected + boundary tags）
  • ✓ @nx/nuxt plugin
  • ✓ nx graph（互動式，可匯出）
  • ○ NX Cloud 視需要（distributed cache）
